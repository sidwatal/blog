Arrays and Hashes

When a Simple Variable Just Isn't Enough

Feb 18, 2015

Variables are really important to Ruby. Obviously. They are probably the main way we manipulate data. But once your programs start becoming a little more complicated you realize that it is really hard to replicate the real world using just variables. As powerful as they are, variables can only hold one piece of data at a time. Great, if that is what you need. Not so great if what you are trying to manipulate or represent is a list of things like names of people in your Dev BootCamp cohort or items on a menu. You could try doing something like:

person1 = "Sid"
person2 = "Nancy"
person3 = "John"
person4 = "Mary"

Not that bad if your cohort just has 4 people in it. But even at this point you can see how much redundancy we are generating in our code. The code for each line is almost the same with the exception of the name itself. This is where arrays come in. An array would allow you to add all of the names into a single array variable. So your above code would look like:

people = ["Sid", "Nancy", "John", "Mary"]

Much simpler right. Imagine how much quicker it would be if there were 10 people in your cohort ... 20 people ... 100 PEOPLE.

You will need to keep a couple of things in mind when dealing with arrays. First, the length of the array is the amount of elements in that array. So:

puts people.length
----------
4

Second, the index or position of the first element in the array is [0]. And the index of the last element, "Mary", in our example people array is 3. This may not seem intuitive at first but it is something you will get used to(or so I'm told. Saying something is in the zero-th position just doesn't flow off my tongue ... yet). In our Ruby code this will look like

puts people[0]
puts people[1]
puts people[2]
puts people[3]
----------
Sid
Nancy
John
Mary

The length method is just one array method. You can read all the different methods in the Ruby Docs. If you look at the above code example you will see the return of our mortal enemy, code repetition. The 4 lines of our ruby program are like 80 percent repetition. Again, how annoying would this be if there were many more people in your DBC cohort and all you wanted to do was to print their name to the screen. This brings us to the each method. The each method allows us to iterate over each(see how intuitive Ruby is) element of our array and run it through a block of code. It'll make more sense with the following example.

people.each do |person|
  puts person
end
----------
Sid
Nancy
John
Mary

Lets take each part of our program and see what it does

  1. people - This is telling ruby that the array we want to iterate over is the people array.
  2. .each - We would like ruby to take each element of the array and hold it while something happens to it.
  3. do - This marks the beginning of the code we would like to happen to the array.
  4. |person| - This is probably the hardest thing to grasp conceptually(I know it was for me). Whatever we put between the vertical bars will be a temporary variable each element of the array will be passed to(It could have been called anything. I just chose to call it "person" for ease of reading our program). Once our code starts running, element [0] of the array will become "person", our block of code will be run on it, then element [1] will become "person", and so on. This will happen until our code has been run on all the elements of the array. (If you're still with me, the rest is easy).
  5. puts person - This is our code block. Once the an element of the array has been passed to our person variable, this is the code that acts on it.
  6. end - This ends our code block. Simple.
That was a long explanation, but once you get it, you can see how much simpler it is than having to write code to print each element in the array to the screen.

Arrays are great if all you want is a list of elements but what if you wanted to do something a little more complicated like classify the people in your cohort by which city they are going to attend for the DBC On-site. For this we can use a hash. A hash is very similar to an array but where an array is indexed by a number(say it 5 times, starting at the 0-th element), a hash is organized using a system called a key-value pair. So a hash of a DBC cohort could look like:

my_dbc_hash = {
"Sid" => "New York City",
"Nancy" => "Chicago",
"John" => "San Fransisco",
"Mary" => "San Fransisco"
}

What I have done here is created a hash called my_dbc_hash. Each element has 2 parts to it. The part to the left of the => is called the key. All keys in your hash have to be unique. That makes sense, doesn't it? In our example it isn't really possible for anyone to take the DBC On-site portion in multiple cities. The part to the right of the => is called the value. Multiple keys can have the same value. Still makes sense in our example. Just remember that whereas in arrays you would call elements by their index(people[0]) in hashes you call elements by their key(my_dbc_hash["Sid"]). I'm sure you can figure lots of ways to use hashes that make sense for your programs.

best_beers = {
"Dogfish Head 90 Minute" => "IPA",
"Guinness" => "Stout",
"Lagunitas" => "IPA",
"Newcastle" => "Brown Ale"
}

Enjoy your beer ... I mean arrays and hashes.
